# توثيق تنفيذ المرحلة الثانية

هذا المستند يشرح بالتفصيل ما تم تنفيذه في المرحلة الثانية من المشروع، مع توضيح الهدف من كل خطوة، والملفات التي تم إنشاؤها أو تعديلها، وكيف ترتبط هذه الأجزاء ببعضها.

## نظرة عامة

الهدف من المرحلة الثانية كان بناء الواجهة الأساسية للتطبيق، مع التركيز على أربع نقاط رئيسية:

- إنشاء نظام تصميم مخصص باستخدام `Custom CSS` فقط.
- بناء `Layout` أساسي قابل لإعادة الاستخدام لصفحات المشارك.
- إنشاء `Header` بسيط يدعم التنقل واللغة والخروج.
- تنفيذ شاشة دخول مستقلة تعمل عبر `fetch` بدون إعادة تحميل الصفحة.

تم الالتزام في هذه المرحلة بعدم استخدام أي CSS framework أو JavaScript framework. التنفيذ اعتمد على:

- Blade
- Vanilla JavaScript
- Custom CSS
- Vite بشكل بسيط فقط لتحميل الأصول

## 1. نظام التصميم

### ما الذي تم عمله

تم استبدال الملف `resources/css/app.css` بالكامل ليصبح ملف التصميم الأساسي للمشروع بدل الإعداد الافتراضي السابق.

### لماذا تم ذلك

الملف القديم كان مرتبطًا بمسار Tailwind الافتراضي، بينما الخطة تشترط بوضوح عدم استخدام أي framework للواجهة. لذلك كان من الضروري بناء طبقة CSS جديدة من الصفر تكون:

- هادئة بصريًا
- سهلة التعديل لاحقًا
- مناسبة للغة العربية والإنجليزية
- قابلة لإعادة الاستخدام في بقية المراحل

### ما الذي يحتويه `app.css`

الملف أصبح يحتوي على:

- متغيرات ألوان رئيسية داخل `:root`
- متغيرات للخطوط والمسافات والزوايا والظلال
- طبقة reset/basic styles للعناصر العامة
- أنماط مشتركة مثل:
  - `.container`
  - `.surface-card`
  - `.button`
  - `.button-secondary`
- أنماط الـ header
- أنماط شاشة الدخول
- أنماط responsive للشاشات الصغيرة

### ملاحظات تصميمية

- الألوان اختيرت لتكون هادئة وقريبة من روح الخطة، لكن بدون صلابة نهائية حتى يسهل تعديلها لاحقًا.
- تم استخدام `bg-pattern.svg` كخلفية بصرية بسيطة بدل الدخول مبكرًا في أصول معقدة.
- تم تجهيز قواعد عامة تخدم بقية الصفحات لاحقًا بدل الاقتصار فقط على شاشة الدخول.

## 2. ملفات CSS المساندة

### `resources/css/animations.css`

تم إنشاء هذا الملف ليحتوي على الأنيميشن الأساسية المطلوبة في هذه المرحلة:

- `shake`
- `fade-in`
- `slide-up`

### لماذا تم فصله

فصل الأنيميشن في ملف مستقل يجعل:

- الصيانة أسهل
- التوسع في المراحل القادمة أوضح
- ملف `app.css` أقل ازدحامًا

### `resources/css/rtl.css`

تم إنشاء هذا الملف لمعالجة الفروقات بين `RTL` و`LTR`.

الهدف منه كان عدم خلط كل تفاصيل الاتجاهات داخل ملف التصميم الرئيسي، خصوصًا أن المشروع ثنائي اللغة ويعتمد على تغيير `dir` ديناميكيًا من JavaScript.

### ما الذي يغطيه

- اتجاه بعض مناطق الـ header
- محاذاة النصوص والنماذج
- تحسين سلوك بعض العناصر عند تبديل اللغة

## 3. الخلفية البصرية

### `public/images/bg-pattern.svg`

تم إنشاء ملف SVG بسيط يعمل كخلفية زخرفية خفيفة مستوحاة من النمط الهندسي العام المطلوب في الخطة.

### سبب استخدام SVG

- خفيف جدًا
- مناسب للخلفيات المتكررة
- سهل التعديل مستقبلًا
- لا يحتاج مكتبات أو ملفات صور كبيرة

## 4. إزالة اعتماد Tailwind من مسار الواجهة

### `vite.config.js`

تم تعديل هذا الملف لإزالة `tailwindcss` plugin من إعداد Vite.

### لماذا

لأن استمرار وجود هذا الـ plugin كان يتعارض مع شرط المرحلة:

- لا Tailwind
- لا CSS frameworks

### النتيجة

أصبح Vite مستخدمًا فقط لتحميل:

- `resources/css/app.css`
- `resources/js/app.js`

وهذا ينسجم تمامًا مع التنفيذ المطلوب في الخطة.

## 5. تهيئة JavaScript المشتركة

### `resources/js/app.js`

تم تحويل هذا الملف من ملف شبه فارغ إلى نقطة تهيئة مشتركة للواجهة.

### ما الذي تم إضافته

- `import './lang';`
- تفعيل زر فتح/إغلاق قائمة الـ header على الشاشات الصغيرة
- تسجيل `service worker` عند تحميل الصفحة

### لماذا تم الإبقاء عليه بسيطًا

الخطة طلبت استخدام Vanilla JS فقط، ومرحلة الواجهة الأساسية لا تحتاج منطقًا معقدًا بعد. لذلك تم الاكتفاء بمنطق واضح ومحدود بدل بناء بنية JS كبيرة مبكرًا.

## 6. الـ Layout الأساسي

### `resources/views/layouts/app.blade.php`

تم إنشاء هذا الملف كـ layout أساسي لصفحات المشارك.

### ما الذي يقدمه

- تحديد `lang` و`dir` على وسم `<html>` بناءً على `session('lang', 'ar')`
- تضمين:
  - `csrf-token`
  - `theme-color`
  - `manifest.json`
- تحميل ملفات CSS وJS عبر `@vite`
- تضمين `components.header`
- توفير مساحة مركزية للمحتوى عبر `@yield('content')`

### لماذا كانت هذه الخطوة مهمة

لأن بقية صفحات المشارك تحتاج أساسًا موحدًا يجمع:

- نظام اللغة
- مظهر الصفحة
- الأصول
- الـ header
- التسجيل البسيط للـ PWA

بناء هذا الـ layout مبكرًا يجعل تنفيذ المراحل اللاحقة أسرع وأكثر اتساقًا.

## 7. مكوّن الـ Header

### `resources/views/components/header.blade.php`

تم إنشاء header reusable مخصص لصفحات المشارك.

### ما الذي يحتويه

- شعار نصي `Rawnaq`
- وصف قصير تحت الشعار
- روابط التنقل الحالية:
  - `morris`
  - `fabrics`
  - `technique`
  - `design-tool`
  - `survey`
- زري تبديل اللغة `AR` و`EN`
- اسم المشارك من الجلسة
- نموذج `POST` لتسجيل الخروج

### قرارات مهمة في التنفيذ

#### 1. استخدام route names الموجودة فعليًا

تم ربط عناصر التنقل مباشرة بالمسارات الموجودة في `routes/web.php` بدل افتراض مسارات جديدة.

#### 2. دعم اللغة بطريقتين

تم تجهيز العناصر بـ `data-lang-key` حتى يعمل عليها `lang.js`، لكن تم أيضًا طباعة نص ابتدائي في Blade بالعربية أو الإنجليزية حتى تكون الصفحة مفهومة منذ أول تحميل وحتى قبل تشغيل JavaScript.

#### 3. دعم responsive بسيط

تمت إضافة زر `header-toggle` مع منطق JavaScript بسيط لفتح وإغلاق القائمة على الشاشات الصغيرة.

هذا ليس تنفيذًا نهائيًا معقدًا للـ mobile drawer، لكنه أساس عملي ومتوافق مع الخطة ويمكن تطويره لاحقًا.

## 8. شاشة الدخول المستقلة

### `resources/views/auth/login.blade.php`

تم إنشاء شاشة دخول مستقلة بالكامل، ولا تعتمد على `layouts.app`.

### لماذا كانت مستقلة

لأن الخطة نصت بوضوح على أن شاشة الدخول يجب أن تكون مستقلة، ولها تصميم خاص مختلف عن بقية صفحات المشارك.

### ما الذي تحتويه

- خلفية داكنة
- بطاقة دخول مركزية
- شعار بسيط `NW`
- عنوان ووصف
- حقل إدخال لكود المشاركة
- زر إرسال
- منطقة رسائل خطأ
- أزرار تبديل اللغة

### دعم اللغة داخل الشاشة

تم استخدام:

- `data-lang-key`
- `data-lang-placeholder`

حتى يصبح النص قابلاً للتبديل عبر `lang.js` بدون reload.

## 9. تفاعل الدخول عبر `fetch`

### أين تم تنفيذ المنطق

تمت كتابة منطق الدخول داخل نفس ملف `login.blade.php` باستخدام JavaScript inline في هذه المرحلة.

### لماذا داخل الصفحة

لأن هذا السلوك مرتبط مباشرةً بشاشة واحدة فقط، ووضعه داخل الصفحة هنا يجعل:

- التنفيذ أسرع
- الربط أوضح
- مراجعة شاشة الدخول أسهل

ويمكن نقله إلى ملف مستقل لاحقًا إذا زاد حجمه.

### ما الذي يفعله الكود

#### 1. يمنع الإرسال التقليدي

تم استخدام:

- `event.preventDefault()`

حتى لا يحصل reload للصفحة عند الإرسال.

#### 2. ينظف الإدخال

تم تقييد الحقل ليقبل أرقامًا فقط، مع قص القيمة إلى 4 أرقام.

#### 3. يرسل الطلب بصيغة JSON

الطلب يرسل إلى:

- `POST /login`

مع الهيدر التالي:

- `Accept: application/json`
- `Content-Type: application/json`
- `X-CSRF-TOKEN`

### لماذا كان `Accept: application/json` مهمًا

لأن `ParticipantAuthController@login()` يعتمد على `wantsJson()` ليقرر هل يعيد JSON أم Redirect/HTML.

بدون هذا الهيدر، كانت شاشة الدخول قد تستقبل استجابة غير مناسبة لتجربة AJAX.

#### 4. يتعامل مع الأخطاء

تم دعم نوعين من أخطاء `422`:

- `payload.error`
- `payload.errors.code[0]`

وهذا يضمن التعامل مع:

- أخطاء منطق التحقق الموجودة داخل المتحكم
- أخطاء validation الافتراضية من Laravel

#### 5. يفعّل تأثير `shake`

عند الخطأ:

- يتم تلوين الحقل كحقل خطأ
- يتم إظهار الرسالة
- يتم تشغيل أنيميشن `shake` على البطاقة

#### 6. يعالج حالة النجاح

عند وجود:

- `payload.redirect`

يتم تحويل المستخدم تلقائيًا إلى الصفحة الأولى (`morris`).

## 10. إنشاء صفحات تمهيدية للمسارات الحالية

بما أن تسجيل الدخول الناجح يوجّه إلى `morris`، وبما أن بقية المسارات موجودة فعليًا في المشروع، تم إنشاء صفحات Blade تمهيدية حتى لا تنكسر التجربة بعد نجاح الدخول.

### الملفات التي أُضيفت

- `resources/views/pages/morris.blade.php`
- `resources/views/pages/fabrics.blade.php`
- `resources/views/pages/technique.blade.php`
- `resources/views/pages/design-tool.blade.php`
- `resources/views/pages/survey.blade.php`
- `resources/views/pages/thank-you.blade.php`

### لماذا أُضيفت

هذه الصفحات ليست تنفيذًا كاملاً لمحتوى المرحلة الثالثة وما بعدها، لكنها كانت ضرورية لسببين:

- منع أخطاء Blade بسبب `view(...)` غير الموجودة
- جعل تدفق المرحلة الثانية قابلًا للاستخدام بعد تسجيل الدخول

بمعنى آخر: تم إنشاء طبقة انتقالية سليمة حتى لا تكون شاشة الدخول معزولة عن بقية التطبيق.

## 11. كيف ترتبط الملفات ببعضها

يمكن تلخيص العلاقة كالتالي:

1. `vite.config.js` يحمّل الأصول الأساسية.
2. `resources/js/app.js` يستورد `lang.js` ويشغّل منطق الواجهة العامة.
3. `resources/views/layouts/app.blade.php` يجمع CSS وJS والـ header.
4. `resources/views/components/header.blade.php` يعتمد على:
   - الجلسة
   - أسماء المسارات
   - `data-lang-switch`
5. `resources/views/auth/login.blade.php` تستخدم نفس الأصول لكن بشكل مستقل، وترسل الدخول عبر AJAX.
6. `ParticipantAuthController@login()` يرجع JSON عند طلبه عبر `fetch` ثم يعيد توجيه المستخدم إلى `morris`.
7. الصفحات التمهيدية تستقبل المستخدم بعد نجاح الدخول وتمنع كسر التدفق.

## 12. ما الذي تم التحقق منه بعد التنفيذ

بعد الانتهاء من التنفيذ تم إجراء تحقق أساسي يشمل:

- مراجعة lints للملفات المعدلة
- التأكد من عدم وجود أخطاء lint ظاهرة
- التحقق من وجود مسارات `login` و`morris` عبر `php artisan route:list`

### نتيجة التحقق

- لا توجد مشاكل lint ظاهرة على الملفات التي تم تعديلها
- المسارات الأساسية المطلوبة موجودة

## 12.1 حل مشكلة Vite manifest (ViteManifestNotFoundException)

### المشكلة

عند فتح صفحة الدخول ظهر الخطأ التالي:

- `Illuminate\\Foundation\\ViteManifestNotFoundException`
- والسبب: عدم وجود `public/build/manifest.json`

هذا يحدث عندما تكون الصفحات تستخدم `@vite(...)` لكن لم يتم تشغيل build للأصول بعد، وبالتالي لا يوجد manifest يُخبر Laravel بمسارات ملفات CSS/JS المولدة.

### ما الذي تم عمله لحلها

1) تم تثبيت الاعتماديات الأمامية:

- `npm install`

2) تم توليد ملفات Vite الإنتاجية والـ manifest:

- `npx vite build`

3) تم التأكد من وجود الملف:

- `public/build/manifest.json`

4) تم مسح الكاش الخاص بالـ views لضمان عدم بقاء أي مخرجات قديمة:

- `php artisan view:clear`

### النتيجة

بعد توليد `manifest.json` لن يعود Laravel يرمي الاستثناء عند استخدام `@vite(...)`، وتبدأ صفحات Blade بتحميل CSS/JS بشكل طبيعي.
